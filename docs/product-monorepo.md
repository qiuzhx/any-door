# any-door 产品与 Monorepo 说明

本文描述 **仓库形态**、**小程序与服务端首期产品范围**，以及 **里程碑顺序**。技术结论汇总见 [`tech-decisions.md`](./tech-decisions.md)。

---

## 1. Monorepo 目录约定（草案）

以下目录名为推荐默认，可在脚手架阶段微调命名（例如将小程序目录改为 `mp-weapp`）。

| 路径 | 职责 |
|------|------|
| **`apps/api`**（或 `apps/server`） | 对外 HTTP API；首期以 **Chat + 流式响应** 为主（如 SSE）。部署目标：**Vercel**。 |
| **`apps/mini-program`**（或 `apps/mp-weapp`） | 微信小程序载体：**Taro + Vite + TailwindCSS**。 |
| **`packages/shared`**（可选） | 共享常量、API 路径前缀、`zod` 请求/响应契约雏形等；首期可为占位包，待契约稳定后再充实。 |

### Turborepo 管线（原则）

- **小程序**：独立一条构建线（WeChat 小程序编译产物）。
- **服务端**：类型检查 / 打包（随 Vercel 构建策略而定）一条线。

两条管线由 **Turborepo** 编排依赖与缓存；任务级 `turbo.json` 细节在脚手架阶段落地，本文不写死具体 JSON。

## 2. 客户端技术栈（首期）

| 层 | 选型 |
|----|------|
| 运行时框架 | **Taro** |
| 构建 | **Vite** |
| 样式 | **TailwindCSS** |
| 目标平台 | **微信小程序**（后续若扩展其它端，可在同一 Monorepo 内增设编译配置）

Tailwind 与 Taro 的集成需在脚手架阶段锁定插件与版本矩阵（本文档不绑定具体 npm 版本）。

## 3. 小程序首期主要内容（MVP）

| 能力 | 说明 |
|------|------|
| AI 对话 | 会话界面（气泡或列表）、输入区与发送动作；首期可先做单会话或不持久服务端会话。 |
| 流式输出 | 消费服务端 **SSE**（或与运行时兼容的 **chunked HTTP / ReadableStream** 等价形态）；助手回复随增量逐步展现。 |
| Markdown 解析与渲染 | 对流式拼接后的正文做 **Markdown** 渲染；首期以实现稳定展示为主。 |

### 微信小程序与「流式」的注意点

- **运行环境与 API 差异**：基础库版本、`wx.request` / `fetch`（若可用）对流式、chunk 的处理与浏览器不完全一致。
- **首期目标**：优先 **打通端到端增量展示通路**；具体采用 SSE、分块 JSON 或兼容层可在实现阶段选定并在本文档后续修订中补充。
- **安全与展示策略（可二期细化）**：外链跳转、代码块复制、HTML 注入风险等，建议在 Markdown 渲染层逐步加固。

## 4. 服务端首期契约（产品视角）

| 项 | 说明 |
|----|------|
| 形态 | **Chat Completions** 语义：客户端发送消息列表（或等价 payload），服务端返回助手回复。 |
| 流式 | 响应支持 **流式**，以便小程序侧逐字/逐段更新 UI。 |
| AI SDK | **未定稿**；服务端封装需满足流式与后续可替换供应商（详见 [`tech-decisions.md`](./tech-decisions.md) 第五节）。 |

### 暂缓（首期不做或仅文案占位）

- **服务端持久化会话历史**（可选用本地草稿或前端缓存过渡）。
- **复杂工具调用 / LangGraph 编排**（与路线图二期对齐）。

## 5. API 路径示例（非最终实现）

以下为便于对齐的示例，最终以 OpenAPI 或共享 `zod` 契约为准：

- **`POST /v1/chat`**：非流式或统一入口（若服务端区分流式可通过 `Accept` / query / 子路径约定）。
- **流式变体**：例如 **`POST /v1/chat/stream`**（SSE）——实现阶段择一并保持小程序与文档一致。

---

## 6. 里程碑顺序（建议）

1. **Monorepo 骨架**：根 workspace + Turborepo；`apps/api` 与 `apps/mini-program` 各自 Hello。
2. **服务端**：`/v1/chat`（或等价路径）与 **流式响应** 打通（可用占位上游或 mock）。
3. **小程序**：请求上述接口，完成 **流式接入 UI**，并接通 **Markdown 渲染**。
4. **产品增强**：错误提示、中止生成、鉴权（API Key / `openId` 等）、观测与限流。
